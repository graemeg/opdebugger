== Overview

The Object Pascal Debug Format (OPDF) provides native, first-class support for Object Pascal concepts that are poorly represented in DWARF:

* *Properties* with getter/setter methods (field-backed and method-backed)
* *String types* (ShortString, AnsiString, UnicodeString)
* *Dynamic arrays* with correct length display
* *Class instances* with inherited fields and interfaces
* *Sets* with enum member name display
* *Method call injection* for getter property evaluation

OPDF data is embedded as an ELF `.opdf` section, with all addresses resolved at link time by the assembler/linker. The FPC compiler emits this section via the `-gO` flag.

== Project Structure

This is a multi-module project built with https://github.com/graemeg/pasbuild[PasBuild]:

* *opdf-lib*: OPDF binary format definitions, reader/writer library, ELF section extractor
* *pdr-core*: PDR debugger core engine (hexagonal architecture, platform-agnostic)
* *pdr-adapters*: Platform-specific adapters (Linux ptrace, OPDF reader, x86_64/i386 arch)
* *pdr-cli*: Command-line debugger interface (REPL)
* *tools*: Diagnostic tools (`opdf_dump` — OPDF section analyser)

== Building

Requires https://github.com/graemeg/pasbuild[PasBuild] and FPC 3.3.1+ with OPDF support (`-gO` flag):

[source,bash]
----
# Build all modules
pasbuild compile

# Build specific module
pasbuild compile -m pdr-core

# View dependency graph
pasbuild compile -v
----

Integration tests use a shell-based test runner that compiles test programs with `-gO`, feeds debugger commands via stdin, and compares output against expected results:

== Current Status

PDR is a standalone CLI debugger (reference implementation) that reads OPDF data and controls the debuggee via Linux ptrace. It currently supports:

* Breakpoints by file:line, hex address, or variable name
* Hit-count conditional breakpoints (`break ... if count=N`)
* Source-level stepping (step over and step into)
* Call stack display with function names and source locations
* Variable evaluation for all Object Pascal types: primitives, floats, enums, sets, ShortString, AnsiString, UnicodeString, pointers, static arrays, dynamic arrays, records, classes (with field display), interfaces
* Compile-time constant evaluation
* Property evaluation: field-backed (automatic) and method-backed (via call injection)
* Array slice display: `print arr[2..5]`
* In-process variable assignment: `set x = 42`, `set day = Monday`
* Local variable listing with scope awareness (including nested procedures)
* Structured inspect command for records, classes, and interfaces
* Auto-print display list (`display`/`undisplay`)
* Hardware watchpoints via x86_64 debug registers (DR0-DR3)
* Break on exception raise with class name and message display
* Method call injection for getter property evaluation (x86_64 SysV ABI, register save/restore, INT3 sentinel, red zone handling, managed return types)

27 automated integration tests cover all of the above.

=== Known Limitations

* ELF64 only (Linux x86_64). Windows PE/COFF and macOS Mach-O are planned.
* Stack unwinding relies on RBP frame pointer chains, which breaks with optimised RTL compiled without frame pointers.
* Variant records, Variant type, and generics are not yet covered.

See `docs/progress.adoc` for the full progress tracker and roadmap.

== Architecture

PDR uses Hexagonal Architecture (Ports and Adapters):

* *Core Domain*: Pure debugging logic, platform-agnostic (`pdr_engine.pas`, `pdr_typesys.pas`)
* *Ports*: Interfaces defined in `pdr_ports.pas` (IProcessController, IDebugInfoReader, IArchAdapter)
* *Adapters*: Platform-specific implementations (Linux ptrace, OPDF reader, x86_64 arch)

This enables:

* Platform independence — core engine has no platform imports
* Support for multiple debug formats (OPDF, potentially DWARF)
* Multiple UI frontends (CLI, TUI, IDE integration) — integrating into an IDE requires only implementing the adapter interfaces; all type evaluation, property resolution, breakpoint logic, call injection, and display formatting are reusable as-is

== Relationship with DWARF

OPDF does not replace DWARF. DWARF remains valuable for interoperability with standard tools (GDB, LLDB, Valgrind, perf), for platforms where OPDF is not yet supported, and for features that OPDF does not yet cover (e.g. register-allocated variables in optimised code, `.eh_frame`-based stack unwinding).

OPDF and DWARF can coexist in the same binary — they use separate ELF sections. A user who needs GDB compatibility continues using `-gw`. A user who wants first-class Object Pascal debugging uses `-gO`.
