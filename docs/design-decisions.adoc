== Overview

This document captures key design decisions for the OPDF (Object Pascal Debug Format) and PDR (Pascal Debug Reference) debugger project, incorporating improvements beyond the initial Gemini discussion.

=== Code Sharing with FPC

**Decision**: Manual copy initially, symlink for development convenience.

**FPC Integration**:

----
freepascal/fpc/                      # FPC repository
├── compiler/
│   ├── dbgdwarf.pas
│   └── dbgopdf.pas                  # OPDF writer (imports shared code)
└── packages/
    └── opdf/                        # Symlink → opdebugger/opdf-lib/
        └── src/main/pascal/
            └── ogopdf.pas           # Symlinked shared definitions
----

**Workflow**:

Development phase:
```bash
# In FPC repo
cd packages/
ln -s /path/to/opdebugger/opdf-lib/src/main/pascal opdf

# In dbgopdf.pas
uses
  {$I ../packages/opdf/ogopdf.pas}
```

Release phase:
```bash
# Copy files into FPC for upstream submission
cp opdebugger/opdf-lib/src/main/pascal/*.pas fpc/packages/opdf/
```

**Rationale**: Symlink allows rapid development without manual copying; copy for clean upstream submission.

== Architecture Pattern

=== Decision: Hexagonal Architecture (Ports & Adapters)

**Core Principles**:

* **Domain Core**: Pure business logic, no platform dependencies
* **Ports**: Interfaces defining contracts (`IProcessController`, `IDebugInfoReader`)
* **Adapters**: Platform/format-specific implementations
* **Dependency Injection**: Compose system at startup

**Key Patterns Applied**:

. **Strategy Pattern**: Type evaluators (`ITypeEvaluator`)
. **Factory Pattern**: Platform controller creation
. **Command Pattern**: CLI commands
. **Adapter Pattern**: Debug info readers, process controllers

**Benefits**:

* **Testability**: Mock adapters for unit tests
* **Platform Independence**: Add FreeBSD/Windows/macOS without core changes
* **Format Flexibility**: Support DWARF alongside OPDF
* **UI Flexibility**: CLI, TUI, IDE integrations share core

**Directory Mapping**:

----
Domain Core (Hexagon):
  pdr-core/src/main/pascal/
    ├── pdr_engine.pas        # Orchestration
    ├── pdr_typesys.pas       # Type evaluation (Strategy)
    └── pdr_symbols.pas       # Symbol resolution

Ports (Interfaces):
  pdr-core/src/main/pascal/
    └── pdr_ports.pas         # All interfaces

Adapters:
  pdr-adapters/src/main/pascal/
    ├── pdr_opdf_adapter.pas      # IDebugInfoReader impl
    ├── pdr_linux_ptrace.pas      # IProcessController impl
    └── pdr_arch_adapters.pas     # IArchAdapter impls

UI Adapter:
  pdr-cli/src/main/pascal/
    └── pdr_main.pas          # Uses ICommandHandler port
----

== 32-bit vs 64-bit Support

=== Decision: Architecture Metadata in OPDF

**Problem**: Original specification didn't account for pointer size differences.

**Solution**: Add architecture fields to `TOPDFHeader`:

[source,pascal]
----
TOPDFHeader = packed record
  Magic: array[0..3] of Char;  // 'OPDF'
  Version: Word;               // Format version
  BuildID: TGUID;              // UUID for binary linking
  TargetArch: Byte;            // NEW: Architecture ID
  PointerSize: Byte;           // NEW: 4 or 8 bytes
  TotalRecords: Cardinal;      // Number of records
  Flags: Cardinal;             // Feature flags
end;
----

**Architecture IDs**:

[cols="1,2,1"]
|===
|ID |Architecture |Pointer Size

|1
|i386 (32-bit x86)
|4 bytes

|2
|x86_64 (64-bit x86)
|8 bytes

|3
|ARM (32-bit)
|4 bytes

|4
|AArch64 (64-bit ARM)
|8 bytes

|5
|PowerPC (32-bit)
|4 bytes

|6
|PowerPC64
|8 bytes
|===

**Architecture Adapter Pattern**:

[source,pascal]
----
type
  IArchAdapter = interface
    ['{45678901-4567-4567-4567-456789012345}']
    function GetPointerSize: Byte;
    function ReadPointer(Address: QWord; out Ptr: QWord): Boolean;
    function GetCallingConvention: TCallingConvention;
  end;

  TArchX86Adapter = class(TInterfacedObject, IArchAdapter)
    function GetPointerSize: Byte;  // Returns 4
  end;

  TArchX86_64Adapter = class(TInterfacedObject, IArchAdapter)
    function GetPointerSize: Byte;  // Returns 8
  end;
----

**Debugger Initialization**:

[source,pascal]
----
// Read OPDF header
OPDFHeader := ReadHeader(BinaryPath);

// Create appropriate architecture adapter
case OPDFHeader.TargetArch of
  1: ArchAdapter := TArchX86Adapter.Create(ProcessController);
  2: ArchAdapter := TArchX86_64Adapter.Create(ProcessController);
  else
    raise Exception.Create('Unsupported architecture');
end;
----

**Testing Strategy**:

* Phase 1 MVP: 64-bit only (x86_64 Linux)
* Phase 2: Add 32-bit support, test both architectures
* Phase 3+: ARM support

== External vs Embedded Debug Info

=== Decision: External .opdf Files for MVP

**Rationale**:

. **Simpler Implementation**
   - No ELF/PE section manipulation
   - No linker integration required
   - Just write file: `program.opdf`

. **Faster Development Iteration**
   - Regenerate debug info without recompiling binary
   - Easier format debugging

. **Natural Migration Path**
   - External format proves the spec
   - Embedding is "just" different I/O later

**MVP Workflow**:

```bash
# Compile program (no special flags yet)
fpc myprogram.pas

# Generate external OPDF (standalone tool)
./opdf_generator myprogram > myprogram.opdf

# Debug
./pdr myprogram  # PDR automatically finds myprogram.opdf
```

**Future: Embedded Support**:

Phase 2+ adds embedded OPDF in `.opdf` ELF section:

```bash
# Compile with OPDF embedded
fpc -gopdf myprogram.pas  # Embeds .opdf section

# PDR automatically checks for embedded first
./pdr myprogram  # Uses embedded .opdf section
```

**Debugger Loader Logic**:

[source,pascal]
----
function TOPDFLoader.Load(const BinaryPath: String): Boolean;
begin
  // Phase 2+: Check for embedded section first
  if HasEmbeddedOPDF(BinaryPath) then
    Exit(LoadEmbedded(BinaryPath));

  // Phase 1: Check for external file
  OPDFPath := ChangeFileExt(BinaryPath, '.opdf');
  if FileExists(OPDFPath) then
    Exit(LoadExternal(OPDFPath));

  // Not found
  Result := False;
end;
----

│         └─ Parse recTypeX records                           │

│ 6. Implement Evaluator (Strategy)                           │
│    └─ pdr-core/src/main/pascal/pdr_typesys.pas              │
│         ├─ Create TTypeXEvaluator : ITypeEvaluator          │
│         └─ Register evaluator                               │
│                                                              │
│ 7. Test End-to-End                                          │
│    ├─ pasbuild test -m pdr-core (unit tests)                │
│    ├─ pasbuild compile (full build)                         │
│    └─ ./tests/integration/run_tests.sh test_typeX           │
│         └─ PASS ✓                                           │
│                                                              │
│ 8. Document & Commit                                        │
│    ├─ Update CHANGELOG.adoc                                 │
│    └─ git commit -m "Add Type X support"                    │
└─────────────────────────────────────────────────────────────┘
----

**Type Implementation Order** (from implementation-plan.adoc):

. **Phase 1**: Integer (LongInt, Boolean)
. **Phase 2**: ShortString, AnsiString
. **Phase 3**: Pointer, Class, Property (field-backed)
. **Phase 4**: Dynamic arrays, Sets, Generics, Property (getter/setter)

**Benefits**:

* Each iteration delivers complete, testable functionality
* Easy to parallelize (multiple contributors work on different types)
* Incremental validation reduces debugging complexity
* Clear progress tracking

== Design Pattern Details

=== Strategy Pattern: Type Evaluators

**Problem**: Different types require different evaluation logic.

**Solution**: Each type has its own evaluator implementing `ITypeEvaluator`.

[source,pascal]
----
type
  ITypeEvaluator = interface
    function Evaluate(const VarInfo: TVariableInfo;
                     Controller: IProcessController;
                     ArchAdapter: IArchAdapter): TVariableValue;
  end;

  TPrimitiveEvaluator = class(TInterfacedObject, ITypeEvaluator)
    // Evaluates Integer, Boolean, Char
  end;

  TShortStringEvaluator = class(TInterfacedObject, ITypeEvaluator)
    // Reads length byte, then characters
  end;

  TAnsiStringEvaluator = class(TInterfacedObject, ITypeEvaluator)
    // Follows pointer, reads heap metadata
  end;
----

**Registration**:

[source,pascal]
----
procedure TTypeSystem.RegisterDefaultEvaluators;
begin
  RegisterEvaluator('Integer', TPrimitiveEvaluator.Create);
  RegisterEvaluator('LongInt', TPrimitiveEvaluator.Create);
  RegisterEvaluator('ShortString', TShortStringEvaluator.Create);
  RegisterEvaluator('AnsiString', TAnsiStringEvaluator.Create);
  // Extensible: users can add custom evaluators
end;
----

**Benefit**: Adding new type = implement evaluator + register. No core changes.

=== Command Pattern: CLI Commands

**Problem**: Multiple commands with different behaviors.

**Solution**: Each command is an object implementing `ICommand`.

[source,pascal]
----
type
  ICommand = interface
    function Execute(const Args: TArray<String>): Boolean;
    function GetHelp: String;
  end;

  TPrintCommand = class(TInterfacedObject, ICommand)
  private
    FEngine: ICommandHandler;
  public
    constructor Create(AEngine: ICommandHandler);
    function Execute(const Args: TArray<String>): Boolean;
    function GetHelp: String;
  end;
----

**CLI Loop**:

[source,pascal]
----
procedure TCLIDebugger.ProcessCommand(const Input: String);
var
  Parts: TStringArray;
  CmdName: String;
  Cmd: ICommand;
begin
  Parts := Input.Split([' ']);
  CmdName := Parts[0];

  if FCommands.TryGetValue(CmdName, Cmd) then
    Cmd.Execute(Copy(Parts, 1, Length(Parts) - 1))
  else
    WriteLn('Unknown command: ', CmdName);
end;
----

**Benefit**: Easy to add new commands without modifying core loop.

=== Factory Pattern: Platform Controllers

**Problem**: Need different `IProcessController` per platform.

**Solution**: Factory function selects implementation.

[source,pascal]
----
function CreateProcessController: IProcessController;
begin
  {$IFDEF LINUX}
  Result := TLinuxPtraceAdapter.Create;
  {$ENDIF}
  {$IFDEF FREEBSD}
  Result := TFreeBSDPtraceAdapter.Create;
  {$ENDIF}
  {$IFDEF WINDOWS}
  Result := TWindowsDebugAdapter.Create;
  {$ENDIF}
  {$IFDEF DARWIN}
  Result := TMacOSMachAdapter.Create;
  {$ENDIF}
end;
----

**Benefit**: Platform detection centralized; main code platform-agnostic.

== Key Improvements Over Initial Design

. **PasBuild Integration**: Professional build system with dependency management
. **Architecture Metadata**: Explicit pointer size and architecture in OPDF header
. **Hexagonal Architecture**: Clean separation, testability, extensibility
. **Strategy Pattern**: Extensible type evaluation system
. **Command Pattern**: Modular CLI command system
. **32-bit/64-bit Support**: Architecture adapters handle pointer size differences
. **External-First Approach**: Simpler MVP, embedded support added later
. **Mock Testing**: Unit tests without requiring real processes
. **Clear Module Boundaries**: Core, adapters, UI separated
