== Overview

This document analyzes the data type sizes used in the OPDF binary format and justifies decisions for future-proofing.

== Summary of Changes

[cols="2,2,2,3"]
|===
|Field |Original |Updated |Reason

|`RecType`
|`Byte`
|`Byte` ✓
|256 record types sufficient

|`RecSize`
|`Word` (64KB)
|`Cardinal` (4GB)
|Large classes/generics can exceed 64KB

|`TypeID`
|`Word` (65K)
|`Cardinal` (4B)
|Large projects with generics need 100K+ type IDs

|`NameLen`
|`Byte` (255)
|`Word` (65K)
|Qualified names can exceed 255 chars

|`Address`
|`QWord`
|`QWord` ✓
|64-bit addresses supported

|`FieldCount`
|`Word`
|`Cardinal`
|Classes can have 1000+ fields
|===

== Detailed Analysis

=== RecType: Byte (Keep as-is) ✓

**Type**: `Byte` = 0..255

**Current Usage**: 13 record types defined

**Future Expansion Estimate**:

* Current: 13 types
* Phase 1-3: +10 types (local vars, scopes, inline functions)
* Generics support: +5 types (type parameters, constraints)
* Advanced features: +15 types (closures, anonymous methods, attributes)
* Reserved: +20 types
* **Total**: ~63 types

**Verdict**: ✓ **Keep as Byte** - 256 types is more than sufficient.

=== RecSize: Cardinal (Change from Word) ⚠️

**Original**: `Word` = 0..65,535 bytes (64KB max)

**Updated**: `Cardinal` = 0..4,294,967,295 bytes (4GB max)

**Problem Scenarios**:

==== Large Class Definition

[source,pascal]
----
type
  THugeClass = class
    Field001: Integer;
    Field002: String;
    // ... 1000 fields
    Field1000: TObject;
  end;
----

**Size Calculation**:

* Each field descriptor: ~25 bytes (TypeID:4 + Offset:4 + NameLen:2 + Name:~15)
* 1,000 fields × 25 bytes = 25,000 bytes
* Plus class metadata: ~100 bytes
* **Total**: ~25KB (within 64KB, but approaching limit)

==== Generic Instantiations with Many Methods

[source,pascal]
----
type
  TGenericCollection<T> = class
    // 200 methods for different operations
  end;

  // Instantiated for 50 different types
  TMyInt = TGenericCollection<Integer>;
  TMyStr = TGenericCollection<String>;
  // ... 48 more instantiations
----

* Each method: ~50 bytes (signature + name + address)
* 200 methods × 50 bytes = 10,000 bytes per instantiation
* 50 instantiations = 500KB total (spread across records)
* But if stored as single class record: **Exceeds 64KB** ⚠️

==== Real-World Example: Lazarus LCL

Looking at `TControl` class in LCL:

* Direct fields: ~50
* Inherited fields (from TComponent, TPersistent): ~30
* Properties: ~100
* Methods: ~150

Total record size estimate: ~15KB

But `TForm` (inherits from TControl):

* Adds ~200 more properties
* Adds ~300 more methods
* **Estimated**: 50KB+ record ⚠️

**Verdict**: ⚠️ **Change to Cardinal** for safety margin.

**Storage Cost**: +2 bytes per record header (acceptable overhead).

=== TypeID: Cardinal (Change from Word) ⚠️

**Original**: `Word` = 0..65,535 types

**Updated**: `Cardinal` = 0..4,294,967,295 types

**Problem Analysis**:

==== Type Explosion with Generics

Each generic instantiation requires a unique TypeID:

[source,pascal]
----
type
  TList<T> = class ... end;

var
  IntList: TList<Integer>;      // TypeID = 1001
  StrList: TList<String>;       // TypeID = 1002
  ObjList: TList<TMyClass>;     // TypeID = 1003
  PtrList: TList<Pointer>;      // TypeID = 1004
  // Each instantiation needs unique TypeID
----

**Calculation for Large Project**:

* Base project: 10,000 units
* Average 20 types per unit = 200,000 base types ⚠️ (exceeds 65K!)
* Generic containers: `TList<>`, `TDictionary<>`, `TObjectList<>`
* If 1,000 different generic instantiations: +1,000 types
* **Total**: 201,000 type IDs needed

**Real Example: Lazarus IDE**

Rough estimates:

* FPC RTL: ~15,000 types
* LCL: ~8,000 types
* IDE units: ~5,000 types
* Generic instantiations: ~2,000
* **Total**: ~30,000 types (within 65K)

BUT: With heavy generic usage (modern Pascal):

* Add `TDictionary<K,V>` with 100 different K,V pairs = +100
* Add `TList<T>` with 500 different T = +500
* Nested generics: `TList<TDictionary<String, Integer>>` = unique type
* **Realistic large project**: 50,000-100,000+ types ⚠️

**Verdict**: ⚠️ **Change to Cardinal** - Word is insufficient for generic-heavy codebases.

**Storage Cost**: +2 bytes per type reference (affects every record).

=== NameLen: Word (Change from Byte) ⚠️

**Original**: `Byte` = 0..255 characters

**Updated**: `Word` = 0..65,535 characters

**Problem Scenarios**:

==== Qualified Names

Pascal allows fully qualified names in debug info:

[source,pascal]
----
System.Generics.Collections.TDictionary<System.SysUtils.String,
  MyApp.DataModels.TCustomerRecord>.TKeyEnumerator
----

Character count: ~120 characters (within 255, but real examples can be longer)

==== Nested Generic Types

[source,pascal]
----
MyFramework.Collections.Generic.TAdvancedDictionary<
  MyApp.Core.Types.TComplexKeyType,
  MyApp.DataAccess.Models.TDetailedValueRecord
>.TInternalKeyValuePairEnumerator.TCurrentItemAccessor
----

Character count: ~200+ characters ⚠️

==== Mangled Names (if stored)

Some compilers store mangled/decorated names:

----
_ZN10MyCompanyApp15DataAccessLayer10Repository18TCustomerRepository28GetActiveCustomersByRegionIdEi
----

Character count: ~100+ characters (common in C++, less in Pascal, but possible)

==== Real-World Edge Case

Debug info for anonymous methods with capture variables:

[source,pascal]
----
MyUnit.$anonymous_method_MyProcedure$captured_var1_Integer_var2_String_var3_TMyLongClassNameHere
----

Could exceed 255 characters with long unit/procedure/variable names.

**Verdict**: ⚠️ **Change to Word** - Prevents truncation of qualified names.

**Storage Cost**: +1 byte per name field (minimal impact).

=== FieldCount: Cardinal (Change from Word)

**Original**: `Word` = 0..65,535 fields

**Updated**: `Cardinal`

**Analysis**:

Classes with 1,000+ fields are rare but possible:

* Auto-generated classes (from database schemas)
* Code generators (ORM frameworks)
* Extreme edge cases (unlikely to exceed 65K)

**Verdict**: ⚠️ **Change to Cardinal** for consistency with RecSize (if a class can have a large record, it likely has many fields).

**Storage Cost**: +2 bytes per record definition.

== Updated Type Definitions

=== Core Structures

[source,pascal]
----
unit ogopdf;

{$mode objfpc}{$H+}

interface

const
  OPDF_MAGIC = 'OPDF';
  OPDF_VERSION = 1;

type
  { OPDF File Header - 32 bytes }
  TOPDFHeader = packed record
    Magic: array[0..3] of Char;  // 'OPDF' (4 bytes)
    Version: Word;               // Format version (2 bytes)
    BuildID: TGUID;              // UUID (16 bytes)
    TargetArch: Byte;            // Architecture (1 byte)
    PointerSize: Byte;           // Pointer size (1 byte)
    TotalRecords: Cardinal;      // Record count (4 bytes)
    Flags: Cardinal;             // Reserved (4 bytes)
  end;
  // Total: 32 bytes

  { Record Header - 5 bytes }
  TOPDFRecordHeader = packed record
    RecType: Byte;      // Record type (1 byte)
    RecSize: Cardinal;  // Payload size (4 bytes) - CHANGED from Word
  end;

  { Type ID - 4 bytes }
  TTypeID = Cardinal;   // CHANGED from Word

  { Name Length - 2 bytes }
  TNameLen = Word;      // CHANGED from Byte
----

=== Type Definitions

[source,pascal]
----
  { Primitive Type }
  TDefPrimitive = packed record
    TypeID: TTypeID;        // 4 bytes (was Word)
    SizeInBytes: Byte;      // 1 byte
    IsSigned: Byte;         // 1 byte
    NameLen: TNameLen;      // 2 bytes (was Byte)
    // Followed by Name (NameLen bytes)
  end;
  // Size: 8 bytes + name

  { Global Variable }
  TDefGlobalVar = packed record
    TypeID: TTypeID;        // 4 bytes (was Word)
    Address: QWord;         // 8 bytes
    NameLen: TNameLen;      // 2 bytes (was Byte)
    // Followed by Name (NameLen bytes)
  end;
  // Size: 14 bytes + name

  { ShortString Type }
  TDefShortString = packed record
    TypeID: TTypeID;        // 4 bytes (was Word)
    MaxLength: Byte;        // 1 byte
    NameLen: TNameLen;      // 2 bytes (was Byte)
    // Followed by Name (NameLen bytes)
  end;
  // Size: 7 bytes + name

  { Pointer Type }
  TDefPointer = packed record
    TypeID: TTypeID;        // 4 bytes (was Word)
    TargetTypeID: TTypeID;  // 4 bytes (was Word)
    NameLen: TNameLen;      // 2 bytes (was Byte)
    // Followed by Name (NameLen bytes)
  end;
  // Size: 10 bytes + name

  { Record/Class Field Descriptor }
  TFieldDescriptor = packed record
    FieldTypeID: TTypeID;   // 4 bytes (was Word)
    Offset: Cardinal;       // 4 bytes (byte offset in structure)
    NameLen: TNameLen;      // 2 bytes (was Byte)
    // Followed by Name (NameLen bytes)
  end;
  // Size: 10 bytes + name

  { Class Definition }
  TDefClass = packed record
    TypeID: TTypeID;        // 4 bytes (was Word)
    ParentTypeID: TTypeID;  // 4 bytes (was Word)
    VMTAddress: QWord;      // 8 bytes
    InstanceSize: Cardinal; // 4 bytes
    FieldCount: Cardinal;   // 4 bytes (was Word) - CHANGED
    NameLen: TNameLen;      // 2 bytes (was Byte)
    // Followed by Name (NameLen bytes)
    // Followed by FieldCount × TFieldDescriptor
  end;
  // Size: 26 bytes + name + fields
----

== Storage Impact Analysis

=== Per-Record Overhead

[cols="2,1,1,1"]
|===
|Change |Old Size |New Size |Overhead

|RecSize (Word→Cardinal)
|2 bytes
|4 bytes
|+2 bytes/record

|TypeID (Word→Cardinal)
|2 bytes
|4 bytes
|+2 bytes/reference

|NameLen (Byte→Word)
|1 byte
|2 bytes
|+1 byte/name

|FieldCount (Word→Cardinal)
|2 bytes
|4 bytes
|+2 bytes/class
|===

=== Example Project Size Calculation

**Small Project** (1,000 types, 500 variables):

* Old format: ~150KB
* New format: ~170KB
* **Overhead**: +20KB (13% increase)

**Large Project** (50,000 types, 20,000 variables):

* Old format: ~7.5MB
* New format: ~9MB
* **Overhead**: +1.5MB (20% increase)

**Verdict**: ✓ Acceptable overhead for future-proofing.

=== Mitigation Strategies

. **External Debug Files**: Overhead only affects debug builds, not production binaries
. **Compression**: Future `Flags` field can enable compression (easily 50%+ reduction)
. **Lazy Loading**: Debugger can load debug info on-demand

== Migration Path

=== Version Strategy

The `Version` field in `TOPDFHeader` allows format evolution:

* **Version 1** (MVP): Uses updated sizes (Cardinal TypeID, etc.)
* **Future versions**: Can add/change fields with backward compatibility

=== Backward Compatibility

No backward compatibility needed for version 1 (it's new format).

Future versions can support multiple formats:

[source,pascal]
----
case Header.Version of
  1: ParseV1Format(Stream);
  2: ParseV2Format(Stream);
end;
----

== Recommendations

=== Immediate Changes (Version 1)

. ✅ Change `RecSize` from `Word` to `Cardinal`
. ✅ Change `TypeID` from `Word` to `Cardinal`
. ✅ Change `NameLen` from `Byte` to `Word`
. ✅ Change `FieldCount` from `Word` to `Cardinal`
. ✅ Keep `RecType` as `Byte`
. ✅ Keep `Address` as `QWord`

=== Future Considerations

. Add compression support via `Flags` field
. Consider variable-length encoding for TypeIDs (common values use 1-2 bytes)
. Profile real-world projects to validate assumptions

== Conclusion

The updated data type sizes provide robust future-proofing:

* **Cardinal TypeID**: Supports 4 billion types (handles generic explosion)
* **Cardinal RecSize**: Supports 4GB records (handles massive classes)
* **Word NameLen**: Supports 65K character names (handles qualified names)
* **Byte RecType**: 256 record types (sufficient for foreseeable features)

The storage overhead (13-20%) is acceptable given:

* Debug info typically stored externally (doesn't affect production binaries)
* Compression can reduce overhead
* Prevents format limitations as language evolves

All changes implemented in OPDF specification version 1.0.
