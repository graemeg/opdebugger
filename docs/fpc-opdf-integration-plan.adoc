= FPC OPDF Integration Plan

This document outlines the plan for integrating the OPDF debug information format into the Free Pascal Compiler (FPC). It is based on the initial analysis and the existing information in `docs/implementation-plan.adoc`.

== 1. Command-Line Option

A new command-line option, `-gO` (or `-gopdf`), will be introduced to trigger the generation of OPDF debug information.

*   **File to modify**: `fpc-compiler/options.pas`
*   **Procedure to modify**: `TOption.Interpret_G_l`
*   **Action**: Add a case for 'O' in the `Interpret_G_l` procedure. This will set the `paratargetdbg` global variable to a new value, `dbg_opdf`, as suggested in the initial analysis.

== 2. Debug Format Registration

The new OPDF format needs to be registered within the FPC system.

*   **File to modify**: `fpc-compiler/systems.inc`
*   **Type to modify**: `tdbg`
*   **Action**: Add `dbg_opdf` to the `tdbg` enumeration.

*   **File to modify**: `fpc-compiler/systems.pas`
*   **Array to modify**: `dbginfos`
*   **Action**: Add a new record to the `dbginfos` array for `dbg_opdf`, specifying its name as 'OPDF'.

== 3. OPDF Writer Implementation

A new unit will be created to handle the generation of the OPDF file. This implementation will reuse the existing `opdf-lib` code.

*   **Action (Pre-requisite)**: The units from `opdf-lib` (`ogopdf.pas`, `opdf_demangle.pas`, `opdf_io.pas`) must be made available to the FPC build process. This will be done by creating symbolic links from `opdf-lib/src/main/pascal/` to a location within the FPC source tree (e.g., `fpc-compiler/`).

*   **New file**: `fpc-compiler/dbgopdf.pas`
*   **Action**:
    *   Create a new class, `TDBGOpdfWriter`, that inherits from `TDbgWriterBase` (likely defined in `fpc-compiler/dbgbase.pas`).
    *   This class will `use` the symlinked `opdf-lib` units.
    *   Implement the FPC debug writer methods (e.g., `WriteType`, `WriteSymbol`, as mentioned in `implementation-plan.adoc`). These methods will act as a bridge, calling the appropriate functions and procedures within the `opdf-lib` units to build the OPDF data structures.
    *   The writer will be responsible for initializing the OPDF structure and writing the final `.opdf` file.

== 4. Compiler Integration

The new OPDF writer needs to be instantiated and used by the compiler when the `-gO` flag is present.

*   **File to modify**: `fpc-compiler/psystem.pas` (as confirmed in `implementation-plan.adoc`)
*   **Procedure to modify**: `compile`
*   **Action**:
    *   Inside the `compile` procedure, add a check for `target_dbg = dbg_opdf`.
    *   If true, create an instance of `TOPDFDebugWriter` and call its methods to emit the debug information during code generation.

== 5. Target Specifics

For the initial implementation, the focus will be on `x86_64` on Linux, as requested. The OPDF generation logic in `TDBGOpdfWriter` will be written with this specific target in mind.

== 6. Testing Strategy

To ensure the FPC integration is successful and does not introduce regressions, a multi-layered testing strategy should be adopted.

=== 6.1. Unit Testing the OPDF Writer

Before performing end-to-end tests, the `dbgopdf.pas` unit should be tested in isolation.

*   **Approach**: Create a dedicated test program within the FPC source tree that directly instantiates `TDBGOpdfWriter`. This program will feed mock `TDef` and `TSymEntry` data to the `WriteType` and `WriteSymbol` methods.
*   **Verification**: The test will verify that the generated `.opdf` file is well-formed and contains the correct records corresponding to the mock input. The existing `opdf-lib` reader can be used for this verification.
*   **Benefit**: This allows for rapid testing of the OPDF generation logic without needing to compile a full program or depending on the rest of the compiler's output.

=== 6.2. Manual End-to-End Testing

This process validates the entire toolchain, from compilation to debugging.

.   **Build Modified FPC**: Compile the FPC compiler that includes the new OPDF writer.
.   **Compile Test Programs**: Use the newly built FPC with the `-gO` (or `-gopdf`) flag to compile the test programs in `tests/integration/`. This should generate both an executable and a separate `.opdf` file for each test.
.   **Execute Debugger Tests**: For each compiled test program, run it through the `pdr-cli` debugger. Use shell redirection to feed the associated `.commands` file as input to the debugger.
    ```bash
    ./pdr-cli/target/pdr ./tests/integration/test_01_primitives < ./tests/integration/test_01_primitives.commands > ./tests/integration/test_01_primitives.actual
    ```
.   **Verify Output**: Compare the actual output (`.actual` file) with the `.expected` file. Any differences indicate a potential regression or a change in behavior.
    ```bash
    diff -u ./tests/integration/test_01_primitives.expected ./tests/integration/test_01_primitives.actual
    ```

=== 6.3. Automated End-to-End Testing

The manual process described above can be automated by extending the existing test scripts.

*   **Script**: Modify `tests/integration/run_tests.sh` or create a new script (e.g., `run_fpc_integration_tests.sh`).
*   **Modifications**:
    1.  Add a configuration option at the top of the script to specify the path to the newly compiled FPC executable.
    2.  Update the compilation step to use this FPC executable with the `-gO` flag.
    3.  The rest of the script, which handles running the debugger and comparing the output, can remain largely the same.
*   **Benefit**: This provides a repeatable, single-command way to run a full regression test suite, making it easy to catch issues as changes are made to the FPC integration or the debugger itself.
